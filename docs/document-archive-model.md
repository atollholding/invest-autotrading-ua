# Модель архіву документів емітента

Дата: 2026-05-03

## Принцип

Одна публікація в архіві = один пакет розкриття інформації.

Пакет може містити:

- основний документ: PDF, DOC, DOCX або RTF;
- підпис основного документа: `.p7s`;
- XML-звіт;
- підпис XML-звіту: `.xml.p7s`;
- інший допоміжний файл, якщо він юридично належить до цієї публікації.

Оригінальні назви файлів після підписання не перейменовувати. Якщо файл має пов'язаний `.p7s`, назва підпису має залишатися прив'язаною до назви підписаного файлу.

Охайним і стабільним має бути slug публікації або назва JSON-запису, а не назва самого файлу.

## Джерело даних

Вихідні записи архіву зберігаються у:

```text
src/content/documents/groups/*.json
```

Це єдина схема і для старого мігрованого архіву, і для майбутніх публікацій через CMS.

Публічний `src/content/documents/manifest.json` не редагується вручну. Його генерує `scripts/build-manifest.mjs`.

## Простий запис публікації

Адміністратор через CMS заповнює тільки змістовні поля:

- `publicationKind` - тип публікації;
- `createdAt` - прихована дата створення запису в CMS;
- `publishedAt` - дата публікації на сайті;
- `reportingYear` - звітний рік, якщо документ стосується року;
- `periodEnd` - дата станом на / кінець періоду, якщо застосовно;
- `files` - файли пакета;
- `status` - `draft` або `published`.

`createdAt` адміністратор не заповнює вручну. Для нових записів CMS ставить його автоматично, а для мігрованого архіву воно заповнене датою `publishedAt`, щоб записи стабільно сортувалися в адмінці.

Приклад для річного звіту:

```json
{
  "publicationKind": "annual-report",
  "createdAt": "2026-05-03T10:00:00.000Z",
  "publishedAt": "2026-05-03",
  "reportingYear": "2025",
  "periodEnd": "2025-12-31",
  "status": "published",
  "files": [
    "/documents/uploads/31106821_2025.docx",
    "/documents/uploads/31106821_2025.docx.p7s",
    "/documents/uploads/ReportNoRNOKPP.xml",
    "/documents/uploads/ReportNoRNOKPP.xml.p7s"
  ]
}
```

Приклад для структури власності:

```json
{
  "publicationKind": "ownership-structure",
  "createdAt": "2026-05-03T10:00:00.000Z",
  "publishedAt": "2026-05-03",
  "periodEnd": "2025-12-31",
  "status": "published",
  "files": [
    "/documents/uploads/Структура власності емітента станом на 31.12.2025.pdf",
    "/documents/uploads/Структура власності емітента станом на 31.12.2025.pdf.p7s"
  ]
}
```

## Типи публікацій

Підтримувані `publicationKind`:

- `annual-report` - річна інформація;
- `organization-structure` - організаційна структура;
- `ownership-structure` - структура власності;
- `special-info` - особлива інформація;
- `shareholders-meeting` - загальні збори акціонерів;
- `protocol` - протоколи;
- `shareholder-info` - інформація для акціонерів;
- `issuer-info` - інша інформація емітента.

## Автоматично згенеровані поля

Build-скрипт сам формує:

- `id`;
- `title`;
- `displayTitle`;
- `type`;
- `displayType`;
- `year`;
- file `id`;
- file `role`;
- `signsFileId` для `.p7s`;
- `filename`;
- `format`;
- `sizeBytes`;
- `checksumSha256`;

Тому адміністратор не має вручну формувати slug, ID, ролі файлів, зв'язки підписів, розміри або checksum.

`createdAt` є службовим полем CMS, а не юридичною датою публікації. Воно використовується для сортування записів у CMS-списку: нові записи показуються вище старих. Публічний архів сайту спирається на `publishedAt`, `periodEnd` і `reportingYear`.

## Advanced Overrides

Для старих архівних записів або нетипових назв можна використати вкладений блок `advanced`:

```json
{
  "advanced": {
    "id": "2024-richna-informatsiia-emitenta-tsinnykh-paperiv-za-2024-rik",
    "titleOverride": "Річна інформація емітента цінних паперів за 2024 рік",
    "displayTitleOverride": "Річна інформація за 2024 рік"
  }
}
```

Це службовий блок ручних перевизначень. Для нових типових публікацій його не відкривати й не заповнювати.

`advanced` не запускає додаткову генерацію і не записує автоматично сформовані поля назад у CMS-запис. Навпаки: якщо вказати `advanced.id`, `titleOverride` або `displayTitleOverride`, build-скрипт використає ці ручні значення замість автоматично згенерованих.

Для мігрованого архіву `advanced.id` зберігає стабільність старих anchor-посилань і URL виду `/documents/#<id>`. Не змінювати його без окремої потреби.

## Дати

`publishedAt` і `periodEnd` не змішувати.

Для документа "станом на 31.12.2025", який опубліковано 03.05.2026:

- `publishedAt`: `2026-05-03`;
- `periodEnd`: `2025-12-31`;
- `reportingYear`: `2025`, якщо документ стосується звітного року.

`year` у публічному архіві формується з `publishedAt`.

## Файли

Production-файли емітента лежать у:

```text
public/documents/**
```

Файли, завантажені через CMS, за замовчуванням потрапляють у:

```text
public/documents/uploads/
```

Старий мігрований архів вже приведений до тієї самої схеми metadata у `src/content/documents/groups/*.json`.

Видалення JSON-запису публікації не означає автоматичного видалення файлів з `public/documents/uploads/`. Після видалення або заміни публікації треба перевірити orphan-файли:

```bash
npm run documents:orphans
```

Якщо список очікуваний і файли справді більше не потрібні, їх можна видалити окремою командою:

```bash
npm run documents:clean-orphans
```

Ці команди враховують і `published`, і `draft` записи. Якщо файл прив'язаний до чернетки, він не вважається orphan.

## Build-скрипт

`scripts/build-manifest.mjs`:

- читає `src/content/documents/groups/*.json`;
- пропускає записи зі `status: draft`;
- перевіряє наявність кожного файлу;
- рахує `sizeBytes`;
- рахує `checksumSha256`;
- визначає ролі файлів за розширеннями;
- зв'язує `.p7s` з підписаним файлом;
- генерує `src/content/documents/manifest.json`;
- генерує `public/_redirects` зі старих URL, описаних у `src/content/documents/redirects.json`.

Build має зупинитися, якщо файл відсутній, checksum не збігається у verify-режимі або підпис не має цільового файлу.

Для локальної роботи після збереження документа в CMS можна вручну перегенерувати manifest:

```bash
npm run manifest
```

На Cloudflare Pages це робить production build через `npm run build`.

## CMS

Sveltia CMS має редагувати:

- записи публікацій у `src/content/documents/groups/*.json`;
- файли нових публікацій у `public/documents/uploads/**`.

У CMS список публікацій за замовчуванням сортується за прихованим `createdAt` у спадному порядку.

Згенерований `src/content/documents/manifest.json` не редагувати вручну.

## Безпека публікації файлів

Драфт у CMS означає "не показувати запис у публічному архіві", а не "приватно зберігати файл". Чернетки та їхні файли також коммітяться в GitHub і можуть потрапити у build.

Не завантажувати в CMS файли, які не можна публікувати. Якщо файл уже потрапив у GitHub або Cloudflare Pages, його треба вважати опублікованим. Для справді чутливих матеріалів може знадобитися очищення Git history і purge кешу Cloudflare.

Для цього сайту очікувана модель проста: у CMS завантажуються тільки документи емітента, які мають бути публічними, а orphan-перевірка використовується як регулярний контроль після видалень або помилкових завантажень.
