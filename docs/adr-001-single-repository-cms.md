# ADR 001: Єдиний репозиторій для коду, контенту і документів

Дата: 2026-05-03

## Статус

Прийнято.

## Контекст

Сайт є статичним лендінгом ПрАТ "АВТОТРЕЙДІНГ ІНВЕСТ" з невеликою кількістю сторінок, CMS-контентом і публічним архівом документів емітента.

Очікувана операційна модель:

- один основний адміністратор сайту;
- зміни контенту і документів виконуються через Sveltia CMS;
- документи емітента мають публікуватися без зайвої технічної участі;
- код, дизайн і build-логіка змінюються тільки розробником;
- історія Git має залишатися джерелом аудиту і можливості відкату.

## Рішення

Використовувати один GitHub-репозиторій для production-коду, CMS-контенту і публічних документів емітента.

CMS має редагувати тільки визначені content/files collections:

- `src/content/site/**`;
- `src/content/pages/**`;
- `src/content/documents/groups/**`;
- `public/documents/uploads/**`;
- `public/images/uploads/**`.

Для CMS-доступу використовується окремий GitHub-акаунт або службовий акаунт компанії з fine-grained token, обмеженим конкретним репозиторієм і мінімально потрібними permissions:

- `Contents: Read and write`;
- `Metadata: Read`.

Не надавати CMS-токену permissions для administration, secrets, actions, workflows або інших можливостей, які не потрібні для редагування контенту.

## Обґрунтування

Два репозиторії або Git submodule краще відокремлюють код від контенту, але для цього сайту така модель створює зайву операційну складність:

- сайт невеликий;
- адміністратор один;
- документи вже є публічними матеріалами розкриття інформації;
- контентні зміни мають бути простими для нетехнічного адміністратора;
- CMS UI зменшує ризик випадкових помилок, бо адміністратор бачить тільки потрібні форми, поля і файли;
- усі зміни залишаються в Git history і можуть бути відкочені.

Єдиний репозиторій є прагматичним компромісом для поточного масштабу проєкту.

## Обмеження

GitHub fine-grained token з `Contents: Read and write` не обмежує запис окремими шляхами всередині репозиторію. Технічне обмеження на рівні CMS UI не є повноцінною path-level permission моделлю.

Водночас CMS UI є практичним захистом від випадкових помилок: адміністратор працює з формами для контенту і документів, а не з файловим деревом коду.

Тому ця модель покладається на:

- обмежений CMS UI;
- мінімальні token permissions;
- одного відповідального адміністратора;
- Git history і можливість відкату;
- розділення операційної відповідальності між адміністратором контенту і розробником.

## Коли переглянути рішення

Повернутися до моделі з окремим content/documents repository або submodule, якщо:

- з'явиться кілька адміністраторів із різними ролями;
- потрібне path-level обмеження прав;
- документи почнуть проходити складний review workflow;
- зросте кількість сайтів, мовних версій або пов'язаних публікацій;
- з'явиться вимога незалежного журналу аудиту для документів.

## Наслідки

Переваги:

- простіша CMS-експлуатація;
- простіший Cloudflare Pages build;
- один repository як джерело правди;
- простий відкат через Git.

Компроміси:

- немає технічного path-level sandbox для CMS-токена;
- потрібна дисципліна адміністратора;
- критичні зміни коду мають контролюватися branch protection, CODEOWNERS і review-процесом.
